# View-support logic stays in `views/`, but framework-free and tested

A pure transformation that exists only to drive **one** view ‚Äî it has no domain vocabulary a Flow expert would recognise ‚Äî lives next to that view under `views/`, not promoted to a top-level concept folder. But it is extracted out of the React component into a plain-TypeScript module so it is testable without mounting React, exactly as ADR-0001's `QueryRunner` is drivable without a renderer. This is the mirror of ADR-0003: that ADR pulls *domain* logic *out* of views into concept-named modules (`flows/`, `query/`); this one keeps *presentation* logic *in* views, but holds it to the same framework-free, tested bar. The first instance was `views/exportText.ts` (`FlowRecord[]` ‚Üí text, with `exportText.test.ts`); the case that made the rule explicit is `views/graph/`, which lifts ~400 lines of graph computation out of `GraphView.tsx`.

`views/graph/` owns the network-diagram rendering of a Run's Flows, split along its two distinct budgets:

- **`model.ts`** ‚Äî `buildGraph(rows, { maxNodes }) ‚Üí { nodes, edges, ipTotals, hostTotals, droppedForCap }`. The **node-count budget**: which Flows fit under `maxNodes`, edge aggregation, per-IP and per-host totals. Viewport-independent ‚Äî depends only on `[rows, maxNodes]`. `Node` and `Edge` are geometry-free; `Edge` carries the semantic `dircat: 'bi' | 'uni'`, never a colour.
- **`layout.ts`** ‚Äî `layoutGraph(model, { width, height }) ‚Üí { nodes: PositionedNode[], edges, droppedForSpace }`. The **spatial budget**: which nodes physically fit at the current size with `MIN_IP_GAP` spacing, plus their positions. `PositionedNode extends Node` with `x`/`y`/`radius`, so "geometry exists only after layout" is type-enforced.

`GraphView.tsx` keeps only React concerns: two memos (`model` keyed on `[rows, maxNodes]`, `layout` keyed on `[model, width, height]`), edge colour derived at render from `dircat` against the live palette, hover/click wiring, the ResizeObserver and theme-observer, and the SVG drawing.

## Considered Options

- **Promote to a top-level `src/graph/` concept, sibling of `flows/` and `query/`** ‚Äî rejected: ADR-0003 reserves top-level folders for concepts a domain expert names. The CONTEXT.md vocabulary is the test ‚Äî experts talk about Flows, Runs, Hosts, Interfaces, Run Totals; they do not talk about *nodes, edges, host-bubbles, or a node-budget*. Those are visualisation terms ‚Äî one way of *looking at* Flows, like the table or the export. A top-level `graph/` would be a layer/format folder masquerading as a concept, the same mistake ADR-0003 rejected `domain/` and `state/` for. The graph is a *view of* Flows, so it lives under `views/`.
- **Leave the computation inline in `GraphView.tsx` (status quo)** ‚Äî rejected: 1132 lines with the entire model+layout locked in a single `useMemo`, untestable without mounting React. The spacing constraint solver (`distributeGrouped`) is exactly the off-by-one-prone code a test surface exists to protect, and it had none. This is the friction the change removes.
- **Extract but keep it a single `views/graphModel.ts` file with one `buildGraph(rows, { maxNodes, width, height })`** ‚Äî rejected: fuses the two budgets the code actually has. The node-count budget is viewport-independent; folding `width`/`height` into one function means every resize re-runs sorting, edge aggregation, and totals ‚Äî work that did not change ‚Äî and forces every non-spatial test to supply a viewport. The model/layout split lets the model memo stay stable across resize and be asserted without dimensions.
- **Return a ready-made `infoMsg` string from the module** ‚Äî rejected: bakes UI copy into the framework-free core (the ADR-0003 line ‚Äî strings are the view's job; `exportText.ts` is the lone exception because the string *is* its product) and fuses the two budgets into one sentence. The module returns structured counts (`droppedForCap`, `droppedForSpace`); the view composes the message.

## Consequences

- The graph model and layout become the codebase's next pure test surface, matching `flows/*.test.ts`: `model.test.ts` asserts budget enforcement, edge aggregation, and totals; `layout.test.ts` asserts the `distributeGrouped` spacing invariants and spatial pruning.
- Two memos with disjoint dependencies replace one. Resizing re-runs only `layoutGraph`; a theme flip re-runs neither (colour moved to render), where today both re-ran the full aggregation because `palette.edge`/`palette.edgeHighlight` sat in the memo deps.
- Today's `infoMsg` conflates the two budgets ‚Äî it reads `nodeMap.size` *after* spatial pruning but compares against the node-count-budget `included.length`, so nodes dropped for *spacing* are mis-attributed. The split makes each budget a separate count and the bug *fixable*; the extraction itself is behaviour-preserving (the view reproduces today's exact message from `droppedForCap`), and actually surfacing `droppedForSpace` is left as a deliberate follow-up so the move stays verifiable as same-pixels-moved-code.
- Edge geometry (the per-edge B√©zier/arc/arrow math still inlined in the render) is **not** moved here; it is render-time presentation and remains a separate future extraction. `views/graph/` owns the model and the layout, not the drawing.
- The precedent is now explicit: a future pure view-support module (another export format, a different chart) follows `exportText.ts` / `views/graph/` ‚Äî extracted, framework-free, tested, under `views/` ‚Äî and does **not** claim a top-level folder unless it earns a concept name in CONTEXT.md.
