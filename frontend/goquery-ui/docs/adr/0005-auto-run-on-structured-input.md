# Query Runs fire automatically on structured input; free-text still needs an explicit Run

A Run starts on its own, after a ~400ms debounce, whenever the user changes a *structured* parameter — the time range, the attributes multi-select, `sort_by`, `sort_ascending`, or `limit`. These controls commit on a single click (or settle a number), so the change itself is the intent and a separate "Run" press is friction. The three *free-text* parameters — `query_hosts`, `ifaces`, `condition` — are excluded: they are typed character-by-character and have no unambiguous "done" moment, so they commit explicitly (Enter runs immediately; blur commits the value but leaves the Run control *dirty* and fires nothing). The condition field keeps its existing trailing-space → `validate()` behaviour for live feedback; only Enter promotes that to a Run. This is one policy across both execution modes: the debounced auto-run applies to streaming and non-streaming alike, retiring the streaming-is-manual asymmetry from ADR-0001.

The Run control is a single morphing button in the header slot beside the TimeRangePicker, never a second button. It has four states driven by run phase and a `dirty` flag (current free-text params ≠ the last-run snapshot): **clean** — quiet/secondary, always enabled, acts as Refresh (re-executes current params, which pulls fresh data for relative ranges like "Last 5m"); **dirty** — primary/highlighted, signalling unapplied free-text edits; **running under ~450ms** — the "Run" label, disabled, with an inline spinner; **running over ~450ms** — morphs to a red Cancel (`runner.cancel()`). Cancel therefore surfaces only for genuinely long Runs, and fast auto-runs never flicker the slot.

## Considered Options

- **Auto-run everything, including free-text, and remove the Run button** — rejected: free-text is typed, so debounced auto-run would execute half-written conditions and partial host lists, churning the backend and surfacing transient validation errors mid-keystroke. Neither Enter nor blur reliably means "I'm finished." A control that runs on every settle is wrong for inputs that have no clean settle point.
- **Keep a manual Run for all parameters (status quo)** — rejected: this is the friction the change exists to remove. Structured controls have an unambiguous commit (a click), so gating them behind a second press buys no safety, only a round-trip through the user's hands.
- **Keep streaming manual, preserving ADR-0001's asymmetry** — rejected: behaviour that silently depends on a mode toggle the user may not have noticed is a worse surprise than the SSE-connection churn the asymmetry was meant to avoid — and the debounce plus the runner's generation-based supersession already bound that churn to one live stream at a time.
- **Auto-run per change with no debounce, leaning on supersession** — rejected: selecting four attributes would fire four Runs, each aborting the last, with loading state flickering through the burst. The debounce collapses a burst of structured edits into a single Run; supersession remains the backstop for edits that outrace it.
- **Separate Run and Cancel buttons, or Cancel parked by the progress indicator** — rejected: two slots, or a Cancel far from where the user's controls and attention already sit. One morphing control keeps the affordance in place, and the delayed reveal is what makes a single slot viable under frequent fast auto-runs.

## Consequences

- The param-change effect must be scoped to the structured subset; free-text edits still flow into `params` (and the URL) but must not trigger the effect. This is the core refactor — today *any* `params` change auto-runs (in non-streaming) via `App.tsx`'s effect.
- `dirty` is computed against a `lastRunParams` snapshot taken when a Run starts and is clean again once it completes — so a free-text edit that lands *during* a Run leaves the control dirty afterward.
- Enter in a free-text field bypasses the debounce and runs immediately; the debounce is for structured changes only.
- Cancel is reachable only after the ~450ms threshold; sub-threshold Runs show the inline spinner on a disabled Run. Long queries are the only ones that expose Cancel, which is exactly the "let me cancel long-running queries" intent and nothing more.
- A clean Run is a real Refresh: re-executing identical params is meaningful because relative ranges resolve to a newer window each time.
- Auto-run goes through the same preflight as a manual Run, so invalid or empty params surface a validation error instead of hitting the backend. Initial load auto-runs in both modes now (previously non-streaming only).
- ADR-0001's final Consequence ("run-start policy (auto-run on param change, streaming opt-in) remain App concerns") is superseded by this ADR and should be amended to point here; the run-start *policy* is now specified here, while the *mechanism* (AbortControllers, generation counter, snapshots) stays in the runner per ADR-0001.
- `CONTEXT.md` is left untouched: this is interaction policy, not flow-data domain language (Query, Run, Flow), consistent with ADR-0004.
